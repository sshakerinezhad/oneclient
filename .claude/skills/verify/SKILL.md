# Verify Command

You produce verification tests for a plan. You do NOT implement features.

## Input

Read the current plan (`workplan.md`, `.speckit/tasks.md`, or whatever the user points you to). Read `scratchpad.md` for context.

If `__verify__/` already exists, **delete it entirely** before generating anything. Every run starts clean — no stale tests, no partial leftovers. Wipe first, then regenerate everything from the current plan.

## Process

### 1. Decompose

Break the workplan into every individual atomic change. One change = one thing that can be independently verified. List them all.

For each change also consider how it could break the existing system or problems that could arise.

### 2. Write Tests

For each change, write a single runnable test that returns pass (exit 0) or fail (non-zero). No prose. No maybe. Binary.

For each risk or problem that could rise from changes to the whole system also write tests in the same manor.

If the correct test is obvious, write it. If you're unsure what "correct" looks like for a change, ask. Don't guess.

Tests go in `__verify__/tests/`. Each test file is named to match what it verifies.

### 3. Identify Breakpoints

Find natural boundaries in the workplan where a group of changes form a working unit. These are places where it's intelligent to stop and verify the system actually works before continuing.

For each breakpoint, write an integration script in `__verify__/` named `checkpoint_N.py` (or `.sh`). This script tests that everything up to this point works together. If it fails, everything in that block is considered broken and must be fixed before proceeding.

### 4. Output

```
__verify__/
├── tests/          # one test per atomic change
│   ├── 01_whatever.sh
│   ├── 02_whatever.sh
│   └── ...
├── checkpoint_1.py # integration test for first block
├── checkpoint_2.py # integration test for second block
├── task_map.json   # task-to-test binding (auto-generated)
└── run_all.sh      # runs everything in order, stops on first failure
```

`run_all.sh` executes individual tests in order, then runs the relevant checkpoint script at each breakpoint. Exits non-zero on first failure.

### 5. Bind Tests to Tasks

**This step is mandatory.** After generating all tests and checkpoints, bind them to the task file so the executor agent knows exactly what to run and cannot skip verification.

#### 5a. Create `__verify__/task_map.json`

Build a JSON file mapping every task ID to its test files and checkpoint:

```json
{
  "description": "Maps each task ID to its verification tests and checkpoint. Auto-generated by /verify.",
  "mapping": {
    "T001": {
      "tests": ["__verify__/tests/01_whatever.py", "__verify__/tests/02_whatever.py"],
      "checkpoint": null
    },
    "T002": {
      "tests": ["__verify__/tests/03_whatever.py"],
      "checkpoint": "__verify__/checkpoint_1.py"
    }
  }
}
```

Rules for building the mapping:
- Every test MUST map to exactly one task. No orphan tests.
- A task's checkpoint is set only if it is the LAST task before a breakpoint. Otherwise null.
- For tasks with no automated tests (manual verification), set `"tests": []` and add a `"manual"` field describing what to verify.

#### 5b. Inject `**Verify**` into the task file

For every task in the task file (e.g. `tasks.md`), add a `**Verify**` sub-bullet immediately after the task line containing the exact shell command(s) to run its tests.

Format — task with tests only:
```markdown
- [ ] T001 [US1] Do the thing
  - **Verify**: `python __verify__/tests/01_foo.py && python __verify__/tests/02_bar.py`
```

Format — task with tests AND checkpoint (last task in a block):
```markdown
- [ ] T002 [US1] Do the other thing
  - **Verify**: `python __verify__/tests/03_baz.py`
  - **Checkpoint**: `python __verify__/checkpoint_1.py` — MUST pass before proceeding to next phase
```

Format — manual verification (no automated tests):
```markdown
- [ ] T016 Manual verification step
  - **Verify**: Manual — [describe what to check]
```

Chain multiple tests with `&&` so a single command fails fast on first broken test.

#### 5c. Add enforcement header to the task file

Insert this block near the top of the task file, before the first task:

```markdown
> **⛔ MANDATORY VERIFICATION PROTOCOL — READ THIS BEFORE TOUCHING ANY TASK**
>
> Every task below has a **Verify** field with exact test commands from `__verify__/`.
> The full mapping lives in `__verify__/task_map.json`.
>
> **The rules are non-negotiable:**
>
> 1. After implementing a task, run its **Verify** command(s).
> 2. If ANY test FAILS (non-zero exit) → the task is **FAILED**. Fix your implementation. Do NOT output TASK_COMPLETE.
> 3. If a task has a checkpoint → run it AFTER task tests pass. Checkpoint failure = entire block is broken. STOP.
> 4. You may NOT modify, skip, or weaken any test in `__verify__/`. They are **immutable acceptance criteria**.
> 5. A task is COMPLETE only when ALL its verify tests exit 0. No exceptions.
>
> If you claim TASK_COMPLETE without passing verification, you are lying. Don't.
```

This header exists so the executor agent sees the enforcement rules in the same file as the tasks. It cannot be missed.

## Rules

- You never implement features. Tests only (plus the task binding in Step 5).
- Every test must fail before implementation exists. If it passes, it's worthless.
- If you can't automate a check, say so. Write what the human must manually confirm. Don't fake it.
- Match the project's existing test framework when one exists.
- The implementation agent does not write, modify, or skip these tests.
- Every test must map to a task. Every task must have a Verify field. No gaps.
